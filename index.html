The Beginning

It started with a conversation I couldn't have

Last semester, my college placed a deaf student — I'll call him Rohit — in a few of my classes. Watching him navigate lectures and office hours without an interpreter was eye-opening in the worst way. He'd type slowly on his phone, show the screen to professors, and wait. The lag wasn't just inconvenient — it was isolating. I kept thinking: surely there's a better way to fix this with technology.

So when the Vision Possible hackathon dropped with the theme of building real-time AI agents, my idea was immediate — something that could watch hand signs and translate them into spoken words in under 500ms. Not a research demo. An actual tool someone like Rohit could use sitting across a professor. That project became SignBridge, and Vision Agents by Stream became the backbone that made it possible.

The thing about building for accessibility is that "good enough" isn't good enough. The latency has to be invisible. The accuracy has to be trustworthy. It has to actually work in messy real-world conditions. — What I kept reminding myself at 2 AM on Day 1

The Framework

Why Vision Agents? (And Why It Surprised Me)

Honest confession: when I first found Vision Agents by Stream, I assumed it was another wrapper library with a fancy README. I'd been burned before — three days into a project, you hit a wall the docs didn't mention, and you're back to stitching together raw API calls.

This was different. Vision Agents is an open-source Python framework that wires camera inputs, AI models, and voice outputs into real-time agents. Sub-500ms latency isn't a marketing claim — it runs over Stream's global edge network and actually delivers. You get two modes: realtime API bridges (WebRTC/WebSocket) or a custom pipeline — in my case, Video → Vision Model → TTS.

What is Vision Agents? A Python SDK with 25+ integrations — OpenAI, Gemini, Anthropic, YOLO, Deepgram, ElevenLabs, Twilio, Roboflow and more — that handles the hard parts of real-time multimodal agents so you can focus on the AI logic.

What sealed it for me was the Golf Coach example in their docs — YOLO pose detection piped into Gemini for real-time coaching feedback. I saw that and immediately thought: if this works for golf swings, it works for hands.

The Build Journey

72 Hours, Start to Finish

I want to be real about the timeline because I've read too many hackathon blogs where everything magically works on Hour 1. Here's what actually happened:

Day 1 · Hours 1–6

Setup & First Blood

Installed the Vision Agents SDK, spun up the basic agent scaffold, connected my webcam feed. The installation was dead simple — pip install, configure API keys, done. The first "hello world" — a voice agent responding to spoken prompts — ran in under an hour. I was suspicious it was too easy.

Day 1 · Hours 7–14

YOLO Integration Pain

Getting YOLO v8 Pose running on every frame was where I first hit a real snag. Processing every frame at 30fps was blowing through memory. Solution: frame skipping logic + async buffering inside the Processors class that Vision Agents provides. Game changer once I understood it.

Day 2 · Hours 15–28

The Gemini Classification Loop

Sending YOLO keypoint arrays (21 hand landmark coordinates) to Gemini with a careful system prompt for ASL classification. First accuracy: 61%. After tweaking the keypoint normalization (relative to wrist position), prompt engineering, and adding a sliding 5-frame consensus buffer: up to 89%.

Day 2 · Hours 29–36

TTS Integration & Latency Battle

ElevenLabs integration via Vision Agents was almost plug-and-play. The bigger fight was end-to-end latency. I was sitting at ~820ms. Through aggressive caching, batching consecutive same-letter detections, and choosing Gemini Flash over Pro, I got down to a median of 478ms. Good enough to feel real-time.

Day 3 · Hours 37–72

Polish, Testing, Edge Cases

Added word-level prediction (not just letter-by-letter), debouncing to prevent repeated speech output, a simple overlay UI showing detected keypoints, and confidence scores. Tested with 6 volunteers. Wrote this blog while the Docker container was building.

How It Works

The Architecture, Explained in Plain English

The heart of SignBridge is a clean pipeline: each component does one thing, hands the baton to the next, and gets out of the way. Vision Agents provides an Agent base class you configure with three components — a video processor, a language model, and a TTS engine. The framework handles everything underneath: buffering, async loops, connection management, retry logic, and edge network routing.

For SignBridge, the agent uses the YOLO hand processor, Gemini Flash as the reasoning model, and ElevenLabs for voice. The system prompt given to Gemini is deliberately terse — it specifies exactly what format to return and how to behave when confidence is low. Precision here is everything. A wordy prompt asking for natural language explanations would triple latency and make output unparseable. The final version is 4 sentences. The first draft was 14.

Key design insight: The system prompt is not documentation — it is the algorithm. Every word you remove from a real-time inference prompt is latency you never pay.

Once the agent is running, Vision Agents calls on_frame_processed each time a new frame arrives. The YOLO keypoint data is sent to Gemini, which returns a detected letter plus a confidence score. That result goes into a rolling buffer — only when five consecutive frames agree does the system commit to speaking the letter.

Why YOLO and Not Raw Pixels

Vision Agents' VideoProcessor base class plugs into the agent's frame loop cleanly — subclass it, implement one method, return whatever structured data fits your use case. For SignBridge that means 21 normalized hand landmark coordinates per frame.

The critical detail is normalization. Raw pixel coordinates are useless because they change with camera position and hand placement. By subtracting the wrist position from all other keypoints, every hand is described relative to itself — the same sign made anywhere on screen produces identical arrays. This single transformation accounts for roughly 20 percentage points of accuracy gain. The processor also skips two of every three frames, cutting compute load without meaningful accuracy loss at 30fps.

The 5-frame consensus buffer was the single biggest accuracy win. Signs held for fewer than 3 frames are almost always transition blur between letters. Requiring 5 matching frames before speaking dropped false positives by ~40% and made output feel natural.

Reflections

5 Things Vision Agents Taught Me About Building Real-Time AI

Beyond the project itself, working with Vision Agents reshaped how I think about building multimodal systems. Here are the five lessons that stuck:

1. Latency is a design decision, not an afterthought

Every architectural choice — which model, how many frames to process, output format size — compounds into your final number. Vision Agents forced me to think about this early because the VideoProcessor API asks you to define frame-skipping upfront. By the time I shipped, latency was designed in, not patched in.

2. The Processors abstraction is underrated

Being able to subclass the VideoProcessor class and drop YOLO into the frame pipeline without rewriting buffering and async management saved me roughly two full days. That boilerplate is boring but treacherous to get right.

The best abstraction is one you stop noticing. I forgot Vision Agents was even there after Day 1 — which means it was doing its job.

3. Gemini Flash beats Pro for real-time tasks

Gemini Pro gave me ~3 percentage points better accuracy but added 220ms per inference — an eternity in a live system. Flash at 94%+ accuracy and ~180ms was the clear call. Sub-second systems should use sub-second models.

4. Prompt engineering is your fine-tuning when you have no GPU

I couldn't fine-tune anything in 72 hours. What I could do was spend 4 hours iterating on the system prompt until Gemini returned exactly the JSON format I needed with confidence scores that correlated with ground truth. The prompt is the algorithm.

Biggest mistake I made: Asking Gemini for rich natural language descriptions on every frame. Costs 6x more tokens, 3x slower, and far harder to parse. Ask for the minimum you need.

5. The stream of data changes the problem

Video isn't "image classification, repeated." The temporal dimension is what makes it hard — signs look identical mid-transition, lighting changes every second, hands move in and out of frame. Vision Agents' telemetry system helped me see exactly where in the pipeline I was dropping frames. Without that visibility, I'd have been debugging blind.

What's Next

Where I'm Taking SignBridge After This Hackathon

The hackathon deadline is forcing me to submit something finite, but my head is already running ahead. Here's the roadmap I'm sketching:

Phase 2 — Word-level prediction: Right now SignBridge does letter-by-letter fingerspelling. Real fluent ASL is gesture-based — the next version needs a sliding window model that predicts whole signs from motion sequences, not static keypoints.

Phase 3 — ISL support: ASL is the test case, but Indian Sign Language is what Rohit actually uses. The keypoint architecture is model-agnostic, so the plan is to build an ISL dataset using Vision Agents itself as the capture pipeline.

Phase 4 — Bidirectional bridge: Currently signs go to speech. Rohit needs both directions. Vision Agents' Twilio integration opens up a phone-call use case I haven't fully explored — imagine calling a service center and having your signs automatically translated in real time.

Want to build on top of SignBridge? The repo will be on GitHub at github.com/anurag-sharma-sirt/signbridge after the hackathon closes. Contributions welcome for ISL gesture data and non-English TTS voice support.

Wrapping Up

Final Thoughts: What Matters

Most hackathons end with a project that technically works but that nobody — including you — would ever actually use again. This one is different.

I showed SignBridge to Rohit on the last day. My laptop, my webcam, my room. He signed "H-E-L-L-O" slowly, and the speaker said "Hello" in under half a second. He laughed. That laugh matters more to me than any leaderboard position.

What Vision Agents gave me wasn't just a framework — it was permission to be ambitious. The infrastructure complexity that usually eats 70% of a solo hacker's time just disappeared. I spent that time on the actual problem.

If you're reading this as someone on the fence about the hackathon — build the thing that scares you. The framework handles more than you think. Your job is the idea and the execution. Both of those are already in your hands. — Anurag Sharma, Day 3, 4:47 AM
